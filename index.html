Conceitos Fundamentais: Git e GitHub

O que é Git?

Git é um sistema de controle de versão distribuído que permite rastrear mudanças em arquivos e coordenar o trabalho entre múltiplos desenvolvedores. Criado por Linus Torvalds em 2005, o Git mantém um histórico completo de todas as alterações feitas no projeto.

Características principais:

  • Distribuído: Cada desenvolvedor possui uma cópia completa do histórico
  • Velocidade: Operações locais são extremamente rápidas
  • Integridade: Todos os dados são verificados por checksums
  • Branching: Sistema de ramificação leve e eficiente

O que é GitHub?

GitHub é uma plataforma de hospedagem de código baseada na web que utiliza Git para controle de versão. Oferece funcionalidades colaborativas que facilitam o trabalho em equipe e o gerenciamento de projetos.

Principais funcionalidades:

  • Repositórios: Armazenamento seguro de projetos na nuvem
  • Pull Requests: Sistema de revisão e integração de código
  • Issues: Rastreamento de bugs e funcionalidades
  • Colaboração: Ferramentas para trabalho em equipe

Git vs GitHub

Git

  • Sistema de controle de versão
  • Funciona localmente
  • Software livre e gratuito
  • Interface de linha de comando

GitHub

  • Plataforma de hospedagem
  • Baseado na nuvem
  • Serviço com planos gratuitos/pagos
  • Interface web e gráfica

Termos Essenciais

Repository (Repositório)

Container que armazena o projeto completo, incluindo arquivos, histórico e metadados.

Commit

Snapshot das mudanças no projeto em um momento específico, com uma mensagem descritiva.

Branch

Linha independente de desenvolvimento que permite trabalhar em paralelo sem afetar o código principal.

Clone

Processo de criar uma cópia local completa de um repositório remoto.

Fluxo de Trabalho Git/GitHub

Estados dos Arquivos no Git

No Git, os arquivos podem estar em três estados principais:

Working Directory

Área onde você edita os arquivos. Mudanças ainda não rastreadas pelo Git.

Staging Area (Index)

Área intermediária onde ficam as mudanças preparadas para o próximo commit.

Repository

Onde o Git armazena permanentemente as versões dos arquivos após o commit.

Comandos Básicos do Fluxo

Iniciando um Repositório

git init

Inicializa um novo repositório Git no diretório atual.

Verificando Status

git status

Mostra o estado atual dos arquivos (modificados, staging, etc.).

Adicionando Arquivos

git add arquivo.txt ou git add .

Move arquivos para a staging area. O ponto adiciona todos os arquivos modificados.

Fazendo Commit

git commit -m "mensagem do commit"

Salva as mudanças da staging area no repositório com uma mensagem descritiva.

Visualizando Histórico

git log ou git log --oneline

Mostra o histórico de commits do projeto.

Verificando Diferenças

git diff

Mostra as mudanças entre working directory e staging area.

Trabalhando com Repositórios Remotos

Clonando Repositório

git clone https://github.com/user/repo.git

Cria uma cópia local de um repositório remoto.

Enviando Mudanças

git push origin main

Envia commits locais para o repositório remoto.

Baixando Atualizações

git pull origin main

Baixa e integra mudanças do repositório remoto.

Verificando Remotos

git fetch

Baixa mudanças sem integrar automaticamente.

Workflow com Branches

Criando Branch

git branch nome-da-branch

Cria uma nova branch baseada na branch atual.

Mudando de Branch

git checkout nome-da-branch

Muda para a branch especificada.

Criando e Mudando

git checkout -b nova-branch

Cria uma nova branch e muda para ela imediatamente.

Listando Branches

git branch ou git branch -a

Lista branches locais ou todas as branches (locais e remotas).

Fazendo Merge

git merge nome-da-branch

Integra mudanças de uma branch na branch atual.

Deletando Branch

git branch -d nome-da-branch

Remove uma branch local após o merge.

GitHub Flow - Fluxo Recomendado

1. Criar Branch

Para cada nova feature ou correção, crie uma branch específica.

2. Fazer Commits

Adicione commits com mensagens claras e semânticas.

3. Abrir Pull Request

No GitHub, abra um PR para revisar e discutir as mudanças.

4. Revisar Código

Outros desenvolvedores revisam e sugerem melhorias.

5. Fazer Merge

Após aprovação, as mudanças são integradas à branch principal.

6. Deletar Branch

Remove a branch da feature após o merge bem-sucedido.

Pull Requests no GitHub

Pull Requests (PRs) são o coração da colaboração no GitHub. Eles permitem:

  • Revisão de código: Outros desenvolvedores podem revisar suas mudanças
  • Discussão: Comentários e sugestões sobre o código
  • Testes automatizados: Execução de CI/CD antes do merge
  • Histórico: Documentação das mudanças e decisões tomadas

Como criar um PR:

  1. Push da sua branch para o GitHub
  2. Navegar até o repositório no GitHub
  3. Clicar em "Compare & pull request"
  4. Adicionar título e descrição detalhada
  5. Selecionar reviewers e assignees
  6. Criar o pull request

Resolução de Conflitos

O que são Conflitos no Git?

Conflitos ocorrem quando o Git não consegue automaticamente mesclar mudanças de diferentes branches ou commits. Isso acontece quando as mesmas linhas de um arquivo foram modificadas de formas diferentes em branches distintas.

Situação típica de conflito:

  • Desenvolvedor A modifica a linha 10 do arquivo index.html
  • Desenvolvedor B também modifica a linha 10 do mesmo arquivo
  • Ao fazer merge, o Git não sabe qual versão manter

Quando Conflitos Ocorrem

Durante Merge

Ao tentar unir duas branches que modificaram as mesmas linhas de código.

Durante Pull

Quando você tenta baixar mudanças do repositório remoto que conflitam com seu trabalho local.

Durante Rebase

Ao aplicar commits de uma branch sobre outra, linha por linha.

Durante Cherry-pick

Quando você aplica um commit específico que conflita com o estado atual.

Como Identificar Conflitos

Mensagem de Erro

CONFLICT (content): Merge conflict in arquivo.html

O Git informa claramente quais arquivos têm conflitos.

Status do Repositório

git status

Mostra arquivos marcados como "both modified" ou "unmerged paths".

Marcadores no Arquivo

<<<<<<< HEAD
<h1>Título da Branch Atual</h1>
=======
<h1>Título da Outra Branch</h1>
>>>>>>> nome-da-branch

O Git adiciona esses marcadores para mostrar as versões conflitantes.

Como Resolver Conflitos

1. Identificar Arquivos Conflitantes

git status

Lista todos os arquivos que precisam ser resolvidos.

2. Abrir o Arquivo

Use seu editor de código preferido para visualizar os conflitos.

3. Escolher a Resolução

Você pode:

  • Manter apenas a versão HEAD (branch atual)
  • Manter apenas a versão incoming (outra branch)
  • Combinar ambas as versões
  • Escrever uma solução completamente nova

4. Remover Marcadores

Delete todas as linhas com <<<<<<<, ======= e >>>>>>>.

5. Testar a Resolução

Certifique-se de que o código funciona corretamente após a resolução.

6. Adicionar e Commitar

git add arquivo-resolvido.html
git commit -m "resolve: conflito na seção principal"

Marque o conflito como resolvido e finalize o merge.

Ferramentas para Resolução

Editor de Código

VS Code, Sublime, Atom: Muitos editores destacam conflitos e oferecem botões para aceitar mudanças rapidamente.

Git Mergetool

git mergetool

Abre uma ferramenta visual para resolver conflitos (como vimdiff, meld, ou kdiff3).

GitHub Desktop

Interface gráfica que facilita a visualização e resolução de conflitos.

IDEs

IntelliJ, Eclipse: IDEs oferecem ferramentas avançadas de merge com interface gráfica.

Comandos Úteis Durante Conflitos

Abortar Merge

git merge --abort

Cancela o merge e volta ao estado anterior.

Visualizar Conflito

git diff

Mostra as diferenças entre as versões conflitantes.

Ver Histórico do Arquivo

git log --follow arquivo.html

Mostra o histórico de mudanças do arquivo específico.

Continuar Rebase

git rebase --continue

Continua o rebase após resolver conflitos.

Estratégias de Prevenção

Comunicação da Equipe

Converse com a equipe sobre quem está trabalhando em quais arquivos.

Pull Frequente

git pull origin develop

Mantenha sua branch atualizada com as mudanças mais recentes.

Commits Pequenos

Faça commits menores e mais frequentes para reduzir o escopo dos conflitos.

Branches de Curta Duração

Evite branches que ficam muito tempo sem ser mergeadas.

Revisão de Código

Use Pull Requests para revisar mudanças antes do merge final.

Formatação Consistente

Use ferramentas como Prettier ou ESLint para manter formatação uniforme.

Exemplo Prático de Resolução

Antes da Resolução:

<<<<<<< HEAD
<h1>Guia Completo de Git</h1>
<p>Este é um guia sobre controle de versão.</p>
=======
<h1>Manual de Git e GitHub</h1>
<p>Manual prático para desenvolvedores.</p>
>>>>>>> feat/nova-introducao

Após a Resolução:

<h1>Guia Completo de Git e GitHub</h1>
<p>Manual prático sobre controle de versão para desenvolvedores.</p>

Neste exemplo, combinamos o melhor de ambas as versões: o título mais completo da primeira versão com a descrição mais específica da segunda.
