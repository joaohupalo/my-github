Conceitos Fundamentais: Git e GitHub

O que é Git?

Git é um sistema de controle de versão distribuído que permite rastrear mudanças em arquivos e coordenar o trabalho entre múltiplos desenvolvedores. Criado por Linus Torvalds em 2005, o Git mantém um histórico completo de todas as alterações feitas no projeto.

Características principais:

  • Distribuído: Cada desenvolvedor possui uma cópia completa do histórico
  • Velocidade: Operações locais são extremamente rápidas
  • Integridade: Todos os dados são verificados por checksums
  • Branching: Sistema de ramificação leve e eficiente

O que é GitHub?

GitHub é uma plataforma de hospedagem de código baseada na web que utiliza Git para controle de versão. Oferece funcionalidades colaborativas que facilitam o trabalho em equipe e o gerenciamento de projetos.

Principais funcionalidades:

  • Repositórios: Armazenamento seguro de projetos na nuvem
  • Pull Requests: Sistema de revisão e integração de código
  • Issues: Rastreamento de bugs e funcionalidades
  • Colaboração: Ferramentas para trabalho em equipe

Git vs GitHub

Git

  • Sistema de controle de versão
  • Funciona localmente
  • Software livre e gratuito
  • Interface de linha de comando

GitHub

  • Plataforma de hospedagem
  • Baseado na nuvem
  • Serviço com planos gratuitos/pagos
  • Interface web e gráfica

Termos Essenciais

Repository (Repositório)

Container que armazena o projeto completo, incluindo arquivos, histórico e metadados.

Commit

Snapshot das mudanças no projeto em um momento específico, com uma mensagem descritiva.

Branch

Linha independente de desenvolvimento que permite trabalhar em paralelo sem afetar o código principal.

Clone

Processo de criar uma cópia local completa de um repositório remoto.

Fluxo de Trabalho Git/GitHub

Estados dos Arquivos no Git

No Git, os arquivos podem estar em três estados principais:

Working Directory

Área onde você edita os arquivos. Mudanças ainda não rastreadas pelo Git.

Staging Area (Index)

Área intermediária onde ficam as mudanças preparadas para o próximo commit.

Repository

Onde o Git armazena permanentemente as versões dos arquivos após o commit.

Comandos Básicos do Fluxo

Iniciando um Repositório

git init

Inicializa um novo repositório Git no diretório atual.

Verificando Status

git status

Mostra o estado atual dos arquivos (modificados, staging, etc.).

Adicionando Arquivos

git add arquivo.txt ou git add .

Move arquivos para a staging area. O ponto adiciona todos os arquivos modificados.

Fazendo Commit

git commit -m "mensagem do commit"

Salva as mudanças da staging area no repositório com uma mensagem descritiva.

Visualizando Histórico

git log ou git log --oneline

Mostra o histórico de commits do projeto.

Verificando Diferenças

git diff

Mostra as mudanças entre working directory e staging area.

Trabalhando com Repositórios Remotos

Clonando Repositório

git clone https://github.com/user/repo.git

Cria uma cópia local de um repositório remoto.

Enviando Mudanças

git push origin main

Envia commits locais para o repositório remoto.

Baixando Atualizações

git pull origin main

Baixa e integra mudanças do repositório remoto.

Verificando Remotos

git fetch

Baixa mudanças sem integrar automaticamente.

Workflow com Branches

Criando Branch

git branch nome-da-branch

Cria uma nova branch baseada na branch atual.

Mudando de Branch

git checkout nome-da-branch

Muda para a branch especificada.

Criando e Mudando

git checkout -b nova-branch

Cria uma nova branch e muda para ela imediatamente.

Listando Branches

git branch ou git branch -a

Lista branches locais ou todas as branches (locais e remotas).

Fazendo Merge

git merge nome-da-branch

Integra mudanças de uma branch na branch atual.

Deletando Branch

git branch -d nome-da-branch

Remove uma branch local após o merge.

GitHub Flow - Fluxo Recomendado

1. Criar Branch

Para cada nova feature ou correção, crie uma branch específica.

2. Fazer Commits

Adicione commits com mensagens claras e semânticas.

3. Abrir Pull Request

No GitHub, abra um PR para revisar e discutir as mudanças.

4. Revisar Código

Outros desenvolvedores revisam e sugerem melhorias.

5. Fazer Merge

Após aprovação, as mudanças são integradas à branch principal.

6. Deletar Branch

Remove a branch da feature após o merge bem-sucedido.

Pull Requests no GitHub

Pull Requests (PRs) são o coração da colaboração no GitHub. Eles permitem:

  • Revisão de código: Outros desenvolvedores podem revisar suas mudanças
  • Discussão: Comentários e sugestões sobre o código
  • Testes automatizados: Execução de CI/CD antes do merge
  • Histórico: Documentação das mudanças e decisões tomadas

Como criar um PR:

  1. Push da sua branch para o GitHub
  2. Navegar até o repositório no GitHub
  3. Clicar em "Compare & pull request"
  4. Adicionar título e descrição detalhada
  5. Selecionar reviewers e assignees
  6. Criar o pull request
